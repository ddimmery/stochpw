# Training-loop throughput profiling — results

Generated by `benchmarks/bench_training.py`. **No `src/stochpw` code was modified.**
Environment: JAX 0.8.0, **CPU backend** (single `CpuDevice`), Python 3.12.

Metric convention: a *step* = one mini-batch gradient update; *samples/sec* counts
`2 * batch_size` per step (observed + permuted halves). Compilation is excluded
from steady-state numbers (warm-up runs first; every timed region calls
`jax.block_until_ready`).

---

## Headline finding

**The training loop is dispatch/trace-bound, not compute-bound.** The dominant
cost is that `train_step` (`src/stochpw/training/loop.py:19`) is **not**
`@jax.jit`-decorated, so every step re-traces `jax.value_and_grad(loss_fn_total)`
and dispatches each primitive eagerly. A jitted, otherwise-identical step is
**65–170× faster** on the step compute itself.

My pre-profiling guess that the per-step `float(result.loss)` host sync
(`loop.py:205`) was a major cost was **wrong** — the data shows it is negligible
(see §3). This is exactly why we profiled before changing code.

**Key tell:** steps/sec is essentially **flat (~70 linear / ~45 mlp) across n=250

work; instead it is pinned by fixed per-step Python/trace overhead. Wider
covariates (d_x=50, which inflates the `A⊗X` interaction term) barely moves it.

## 2. Per-step phase attribution + ideal jitted reference (median ms/step)

| case                              | batch create | as-is step | jitted step | gain |
|-----------------------------------|-------------:|-----------:|------------:|-----:|
| n=250   bs=62   linear            |        2.36  |     10.44  |       0.085 | 122× |
| n=250   bs=62   mlp               |        2.49  |     17.61  |       0.145 | 121× |
| n=1000  bs=256  linear            |        2.62  |     12.01  |       0.086 | 139× |
| n=1000  bs=256  mlp               |        2.48  |     20.70  |       0.274 |  76× |
| n=10000 bs=256  linear           |        2.57  |     11.97  |       0.070 | 171× |
| n=10000 bs=256  mlp              |        2.43  |     19.69  |       0.275 |  72× |
| n=1000  d_x=50  bs=256  linear   |        2.57  |     12.06  |       0.088 | 137× |
| n=1000  bs=1000 linear           |        2.84  |     11.09  |       0.170 |  65× |

- **`as-is step`** = `train_step()` exactly as the loop calls it (re-traces
  `value_and_grad` every call). ~10–21 ms/step.
- **`jitted step`** = the same math under one `jax.jit` (loss kept on-device).
  ~0.07–0.27 ms/step.
- **`batch create`** = `create_training_batch()` eagerly (`batch.py:10`): two
  `einsum` interaction builds + four `concatenate`s + a `random.permutation`.
  ~2.5 ms/step, eager.

Per-step budget at n=1000/linear: `batch (2.6) + as-is step (12.0) ≈ 14.6 ms`,
matching the measured ~70 steps/s end-to-end. So:

- Jitting **only the step** → per-step ≈ `2.6 (eager batch) + 0.09 ≈ 2.7 ms`
  ⇒ roughly **5× end-to-end**, with eager batch creation becoming the new floor.
- Jitting the step **and** folding batch creation into the compiled region
  (and/or `lax.scan`-ing the inner loop) → approaches the ~0.09 ms regime ⇒
  **tens of ×**, bounded by Python loop + dispatch overhead per iteration.

## 3. Host-sync cost (`float(loss)` every step vs. once) — the corrected hypothesis

200 jitted steps, syncing the loss to host every step vs. accumulating on-device
and syncing once:

| case                  | sync-each | sync-once | ratio |
|-----------------------|----------:|----------:|------:|
| n=250  bs=62  linear  |  0.0181s  |  0.1414s  | 0.13× |
| n=1000 bs=256 linear  |  0.0198s  |  0.0224s  | 0.89× |
| n=10000 bs=256 linear |  0.0186s  |  0.0238s  | 0.78× |
| n=1000 bs=256 mlp     |  0.0560s  |  0.0571s  | 0.98× |

Syncing every step is **not** a bottleneck here — ratios hover around 1×, and
"sync-once" is sometimes *slower* because `jnp.stack(losses)` adds its own work.
**Conclusion: removing the per-step host sync is not where the gains are.**
(This would change on GPU/TPU, where blocking host transfers can stall the
device — worth keeping the loss on-device there, but it is not the CPU win.)

## 4. cProfile of a warm `fit` (n=1000, linear, 20 epochs = 60 steps)

Total 1.878 s. Top of the cumulative-time stack:

```
1.462s (78%)  train_step                       (loop.py:19)
1.297s        value_and_grad_f                 (jax api.py)
1.075s          _vjp / linearize / direct_linearize   <- autodiff TRACING every step
1.026s        loss_fn_total                    (loop.py:56)
0.921s        core._true_bind / bind_with_trace      <- eager primitive dispatch
0.639s        pjit.cache_miss                        <- per-call jit cache misses
```

78% of fit time is inside `train_step`, and nearly all of that is autodiff
**tracing** (`direct_linearize`) plus eager primitive **dispatch** (`core.bind`,
`pjit.cache_miss`) — repeated 60×. A single `@jax.jit` amortizes this to one
trace+compile. The `float(loss)` conversion does not appear in the hot stack,
corroborating §3.

A JAX profiler trace is also written to `/tmp/stochpw-trace` (`--profile`);
open in TensorBoard or Perfetto to see dispatch gaps between tiny kernels.

## 5. Recommended optimizations (for a follow-up implementation plan)

Ranked by expected payoff vs. effort. **All are deferred — not done here.**

1. **`@jax.jit` the step.** Wrap the gradient+update in a jitted function with
   the discriminator/optimizer/loss/regularizer as static or closed-over args.
   Biggest single win (≈ the 65–170× step speedup; ~5× end-to-end on its own).
   Note: `train_step` currently uses boolean-mask indexing
   (`logits[observed_mask]`, `loop.py:64`) which is **not jit-compatible** — the
   observed rows are the static first half, so replace with a slice (the
   benchmark's `build_jitted_step` shows the pattern).
2. **Fold batch creation into the compiled region** (or jit
   `create_training_batch`): pass batch *indices* + key into the jitted step so
   the `einsum`/`concatenate`/permutation run inside XLA. Removes the ~2.5 ms/step
   eager floor that dominates once (1) lands.
3. **`jax.lax.scan` the inner batch loop** (and optionally `lax.fori_loop` /
   `scan` the epochs): collapses Python-level per-step dispatch to a single
   compiled loop. This is what unlocks the "tens of ×" regime.
4. **GPU/TPU-only:** keep the loss on-device and accumulate, syncing once per
   epoch. Not a CPU win (§3) but avoids device stalls on accelerators.

Validation for any of the above: re-run `benchmarks/bench_training.py` and check
that end-to-end steps/sec moves toward the `jitted` column of §2, and confirm
weights/loss are numerically unchanged vs. the current implementation (fixed
`random_state`).
